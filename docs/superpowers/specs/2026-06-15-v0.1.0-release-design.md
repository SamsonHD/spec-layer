# v0.1.0 Release Design

## Goal

Publish Spec Layer's first normal GitHub release as `v0.1.0`, containing all current product work plus the open-source hardening work, with accurate public documentation and a reproducible release gate.

## Release Boundary

The release candidate combines:

- the latest local `main` history;
- the current inbox clearing and queue redesign work;
- the current AI guideline filling and hardening work;
- the open-source cleanup worktree, including security, accessibility, dependency, governance, and documentation changes.

Uncommitted changes are treated as source material, not disposable state. Integration must preserve them, resolve overlaps deliberately, and avoid resetting or overwriting any working tree.

This is a source release. The workspace packages and web app remain private npm packages and are not published to npm. The GitHub release contains the repository source generated from the signed-off release commit.

## Branch And Version Strategy

Create `release/v0.1.0` in an isolated worktree from the latest local `main`. Integrate committed feature branches first, then apply the open-source cleanup changes and remaining uncommitted feature changes. Resolve conflicts in favor of the newest working behavior while retaining the release hardening and public-repository safeguards.

All package manifests use version `0.1.0` where a version is present. The Git tag is `v0.1.0`. No package publication, release automation, or versioning framework is introduced for this first release.

## Public Documentation

The public entry points must describe the product that ships:

- `README.md` covers purpose, capabilities, local-first security boundary, prerequisites, setup, Figma plugin flow, docs application flow, AI configuration, common commands, limitations, roadmap, and governance links.
- `ARCHITECTURE.md` documents current module boundaries, data flow, storage, local authorization, and external services.
- `apps/web/README.md`, `apps/web/.env.example`, `packages/plugin/TESTING.md`, and `spec/SPEC.md` agree with the implemented behavior.
- `CHANGELOG.md` records the initial release in Keep a Changelog style.
- GitHub release notes summarize user-facing capabilities, security constraints, setup, verification, and known limitations.

Obsolete internal audit and planning documents that could mislead contributors are removed from the release root or clearly identified as historical development material.

## Open Source Repository Practices

The release includes an MIT license, contribution guide, security policy, code of conduct, pull request template, structured issue templates, least-privilege CI, and Dependabot configuration.

The Code of Conduct directs enforcement reports to the private contact methods exposed by maintainer [@SamsonHD](https://github.com/SamsonHD). Security vulnerabilities use GitHub private vulnerability reporting rather than public issues.

Generated inbox content, local settings, environment files, caches, build output, editor metadata, and private design-system data are excluded. Release review includes a tracked-file scan for likely credentials and proprietary data.

## Release Verification

The release candidate must pass from a clean installation:

```text
npm ci
npm run lint
npm run typecheck
npm test
npm run build
npm run build:plugin
npm audit --omit=dev
git diff --check
```

Tests cover the integrated inbox and AI workflows as well as local authorization, request limits, ZIP extraction limits, settings permissions, Figma URL validation, plugin authentication, and accessibility helpers.

Manual verification covers the primary local workflow, plugin build/import readiness, inbox review and filing, AI filling with and without credentials, destructive inbox actions, keyboard navigation, and documentation links.

Any unresolved production security advisory, failing required check, committed credential, broken setup instruction, or release-blocking regression prevents tagging.

## GitHub Delivery

After verification:

1. Commit the integrated release candidate on `release/v0.1.0`.
2. Push the release branch.
3. Open a draft pull request into `main` containing the verification evidence and release checklist.
4. Prepare a draft normal GitHub release targeting `v0.1.0` with generated source archives.

The release and tag remain drafts until the pull request is reviewed and merged. Publishing is a separate explicit action; preparing the first release does not bypass review or publish from an unmerged branch.

## Non-Goals

- Publishing packages to npm.
- Rewriting Git history.
- Hosting the local application as a public multi-user service.
- Adding semantic-release, Changesets, or another automated release framework.
- Introducing new product features beyond the current feature branches.
- Publishing the GitHub release before the release pull request is merged.
