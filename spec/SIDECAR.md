# Spec Layer Sidecar Contract

> **Generated structured data that travels beside a Markdown spec.**

Spec Layer component documentation is authored and reviewed as Markdown. A sidecar is an optional JSON file that mirrors a component's Markdown path and stores the deterministic extraction data used by enhanced renderers and machine workflows.

---

## 1. Purpose

The Markdown file is the portable human-readable spec. It remains the canonical document for prose, review, and source control.

The sidecar stores the generated `IntermediateSpec` for the same component. Renderers use it to avoid reparsing lossy Markdown tables and lists when they need structured data such as variant instances, anatomy, props, tokens, layout summaries, extraction gaps, Figma references, or related components.

Sidecars are optional for compatibility. A Markdown document MUST still render and remain useful without a sidecar. When a sidecar is missing, readers SHOULD fall back to Markdown-only rendering and disable only the enhanced experiences that require structured extraction data.

---

## 2. Path Mirroring

For every component Markdown path under the content root, the sidecar path mirrors the same relative path under `.spec-data` and changes only the extension from `.md` to `.json`.

```text
components/button.md
.spec-data/components/button.json
```

Additional examples:

```text
_inbox/button.md
.spec-data/_inbox/button.json

forms/text-field.md
.spec-data/forms/text-field.json
```

The Markdown path and sidecar path MUST use the same resolved slug. If an exporter resolves a filename collision by writing `card-2.md`, its sidecar MUST be `.spec-data/.../card-2.json`.

Readers MUST NOT require a sidecar to exist for every Markdown file. Writers that have structured extraction data SHOULD write both files together. Writers that only receive raw Markdown MAY write the Markdown file alone.

---

## 3. Markdown Independence

A valid component Markdown file must stand on its own:

- It contains the required frontmatter and canonical Markdown sections described in `spec/SPEC.md`.
- It can be imported, reviewed, edited, diffed, and rendered without `.spec-data`.
- It does not contain hidden dependencies on the sidecar for basic comprehension.

The sidecar enhances the Markdown document; it does not replace it.

---

## 4. Enhanced Rendering and Workflows

When present, the sidecar enables workflows that need exact extracted structure:

- interactive variant grids from `variantInstances`;
- anatomy and properties displays from `anatomy` and `props`;
- token inspection from `tokens`;
- Figma preview workflows from `figmaFile`, `figmaNode`, and variant node ids;
- regeneration or AI-drafting workflows that need the original deterministic extraction;
- machine checks that compare structured extraction data across versions.

These workflows SHOULD degrade gracefully when the sidecar is absent.

---

## 5. Sidecar Shape

The sidecar JSON is the generated `IntermediateSpec` object emitted by the extractor.

```ts
interface IntermediateSpec {
  name: string;
  figmaKey: string;
  figmaFile: string;
  figmaNode: string;
  anatomy: AnatomyPart[];
  props: ComponentProp[];
  variants: VariantAxis[];
  variantInstances: VariantInstance[];
  states: string[];
  tokens: TokenRule[];
  related: string[];
  gaps: Gap[];
  layout: LayoutSummary[];
}

interface VariantInstance {
  nodeId: string;
  name: string;
  values: Record<string, string>;
}
```

Sidecars SHOULD be serialized as UTF-8 JSON with stable, human-readable indentation. Consumers SHOULD ignore unknown fields so future extractor versions can add data without breaking older readers.

The sidecar should describe the same component as the Markdown file. In normal generated output, the following values align:

- `name` matches `component.name` in Markdown frontmatter.
- `figmaKey` matches `component.figma_key`.
- `figmaFile` matches `component.figma_file`.
- `figmaNode` matches `component.figma_node`.

---

## 6. Generated Data

Sidecar data is generated by the extractor and should not be manually edited. Human edits belong in the Markdown document.

If the Markdown file is moved or renamed, tooling MUST move or rename the sidecar with it when the sidecar exists. If the Markdown file is deleted, tooling SHOULD delete the matching sidecar.

If the sidecar becomes stale or invalid, tooling SHOULD regenerate it from the source design data. Manual repair is a last resort and should be treated as generated-file maintenance, not normal authoring.

---

## 7. Import and Export Expectations

Manual Markdown artifacts MAY contain only the `.md` file. They are portable Markdown documents and do not necessarily preserve enhanced structured data.

Single-component sidecar-preserving downloads SHOULD be represented as a one-component ZIP bundle, because the sidecar is a sibling file by contract:

```text
<slug>.md
.spec-data/<slug>.json
```

Bulk ZIP exports that have access to `IntermediateSpec` SHOULD include both:

```text
<folder>/<slug>.md
.spec-data/<folder>/<slug>.json
```

Send-to-docs workflows that receive `IntermediateSpec` SHOULD write the Markdown file and matching sidecar into the docs content root.

Manual Markdown imports MAY create Markdown-only inbox entries. ZIP imports that intend to preserve enhanced rendering SHOULD copy both Markdown entries and matching `.spec-data` entries using the path-mirroring rule above.
